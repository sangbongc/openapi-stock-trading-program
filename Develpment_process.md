## Project Overview

이 프로젝트는 한국투자 OpenAPI를 이용하여 Rule-Based 자동매매 프로그램을 구현하는 것을 목표로 시작하였다.

초기 목표는 단순한 매매 알고리즘 구현이 아니라, 실제 API를 이용하여 데이터 수집, 전략 생성, 주문, 체결 확인까지 하나의 흐름으로 연결하는 것이었다.

프로젝트는 기능을 한 번에 구현하기보다 작은 단위부터 검증하면서 점진적으로 확장하는 방식으로 진행하였다.

## API Integration

개발 초기에는 가장 단순한 현재가 조회 API부터 연동하였다.

현재가 조회는

- API 인증 검증
- HTTP 통신
- JSON 파싱
- SQLite 저장

을 한 번에 확인할 수 있는 가장 단순한 기능이었다.

이후 현재가 저장 기능을 기반으로 일봉 데이터 수집 기능을 추가하였다.

## Market Data Pipeline

초기에는 현재가 저장 기능을 구현하여 데이터 저장 구조를 검증하였다.

이후 전략 계산을 위해 일봉 데이터를 별도로 수집하도록 구조를 변경하였다.

현재가와 일봉 데이터를 분리한 이유는 데이터의 성격이 다르기 때문이다.

현재 프로젝트에서는 일봉 데이터를 주로 사용하지만, 현재가 테이블은 향후 단기 전략이나 실시간 전략을 고려하여 유지하고 있다.

## Strategy Architecture

초기에는 하나의 전략만 고려하였지만, 프로젝트를 진행하면서 여러 전략을 동시에 사용할 수 있는 구조가 필요하였다.

이를 위해

- BaseStrategy
- StrategyFactory
- StrategyEngine

구조를 도입하였다.

새로운 전략은 BaseStrategy를 상속받아 구현할 수 있으며, 기존 TradingEngine을 수정하지 않고 전략을 추가할 수 있도록 구성하였다.

## Order Management

TradingEngine은 매수/매도 여부만 판단하도록 하고, 실제 주문 생성은 OrderManager가 담당하도록 분리하였다.

OrderManager는

- 주문 생성
- API 호출
- 주문 결과 저장
- 예외 처리

를 담당한다.

## Execution Synchronization

프로젝트를 진행하면서 주문과 체결은 서로 다른 이벤트라는 점을 확인하였다.

주문이 접수되었다고 해서 즉시 보유 종목이 변경되는 것은 아니므로, 체결 여부를 별도로 확인하는 ExecutionManager를 추가하였다.

ExecutionManager는

- 주문 상태 조회
- 체결 정보 저장
- PositionManager 갱신

을 담당한다.

또한 프로그램 종료 후 체결이 발생하는 상황을 고려하여 sync 기능을 추가하였다.

## Testing

프로젝트는 기능을 구현할 때마다 테스트를 함께 진행하였다.

주요 검증 과정은 다음과 같다.

- Mock 기반 Unit Test
- Integration Test
- Dry-run 실행
- 한국투자 모의투자 계좌 검증

단계적으로 검증 범위를 확대하면서 기능을 확인하였다.
